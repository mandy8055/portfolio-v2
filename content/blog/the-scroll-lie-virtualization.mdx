## Why Lists Get Slow

When you render a list of 500 items in React, you're not just rendering 500 components. You're creating **500 real DOM nodes**. Each one carries:

- Its own memory footprint
- Layout and style recalculations
- Paint and composite costs
- Scroll event overhead

The browser doesn't care that 480 of those nodes are nowhere near the viewport. It tracks, computes, and manages all of them. Every scroll event. Every frame.

Here's what an innocent-looking list component actually does at scale:

```tsx
// Looks totally fine...
const ResultsList = ({ items }: { items: Item[] }) => (
  <ul>
    {items.map((item) => (
      <li key={item.id}>
        <ResultCard item={item} />
      </li>
    ))}
  </ul>
);

// But at 500 items the DOM looks like:
// <ul>
//   <li>...</li>   вЖР DOM node
//   <li>...</li>   вЖР DOM node
//   ... √Ч 500
// </ul>
// 500 nodes. All alive. All tracked by the browser.
```

This is where the jank comes from. The fix isn't to render a faster card component or optimize your CSS. The fix is to stop creating DOM nodes for things the user isn't looking at.

## The Lie Virtualization Tells

Virtualization's entire premise is this: **only create DOM nodes for items inside the visible viewport.**

If your list can show 8 cards at a time, you maintain roughly 10вАУ12 DOM nodes (a small buffer above and below). The rest simply don't exist in the DOM.

```
Data array (in JS memory): 500 items вЬЕ
Visible area:              items 40вАУ48
DOM nodes alive:           items 36вАУ52 (with buffer)
DOM nodes that don't exist: items 1вАУ35 and 53вАУ500
```

But here's the catch вАФ if you just rendered 12 items, the scrollbar would look completely wrong. It would behave like there are only 12 items total. That breaks the whole experience.

So virtualization has to maintain an illusion. And it does it with two clever tricks.

---

## How The Illusion Works

### Trick 1: The Phantom Container

The outer wrapper gets an explicit height equal to what the **full list would be** if every item was rendered:

```
Total height = itemCount √Ч itemHeight
             = 500 √Ч 72px
             = 36000px
```

This 36,000px invisible box makes the scrollbar behave as if all 500 items are present. The user's muscle memory for scroll distance stays intact.

### Trick 2: Absolute Positioning

The actual rendered items sit **absolutely** inside that phantom container, placed at the exact pixel offset where they should visually appear:

```
Item 40 вЖТ top: 40 √Ч 72 = 2880px
Item 41 вЖТ top: 41 √Ч 72 = 2952px
Item 42 вЖТ top: 42 √Ч 72 = 3024px
```

As the user scrolls, the library runs a tight loop:

1. Read `scrollTop` from the scroll container
2. Calculate which items _should_ be visible
3. Unmount DOM nodes that scrolled out of range
4. Mount DOM nodes for items entering the viewport
5. Position each one at the correct absolute offset
6. User sees nothing unusual. Scroll feels natural.

```
User scrolls (scrollTop = 2900px)
         вЖУ
startIndex = Math.floor(2900 / 72) = 40
endIndex   = startIndex + visibleCount + overscan buffer
         вЖУ
Render items 37вАУ53
Position each at (index √Ч 72)px from the top
         вЖУ
~15 DOM nodes instead of 500
```

This calculation runs on every scroll event, which is why libraries like TanStack Virtual are so carefully optimized вАФ it has to be _fast_.

## An Important Distinction Before We Add Infinite Scroll

There's one thing I want to be very explicit about, because it confused me initially.

**Virtualization destroys DOM nodes. It does NOT destroy data.**

When item 12 scrolls out of view, its `<div>` is removed from the DOM. But the JavaScript object for that item is still sitting in your array, in memory. It never goes anywhere.

```
JS memory (your array) вАФ always intact:
[item1, item2, ..., item12, ..., item42, ..., item500]

DOM вАФ just a snapshot of what's visible right now:
<ResultCard item={item40} />
<ResultCard item={item41} />
<ResultCard item={item42} />  вЖР you're here
<ResultCard item={item43} />
```

So when a user scrolls down to item 200 and then scrolls back up to item 12, **no API call is made.** The data was already fetched. Virtualization just gives it a DOM node again.

This distinction matters a lot when we bring infinite scroll into the picture.

---

## Where Infinite Scroll Fits In

Virtualization manages the **DOM**. Infinite scroll manages the **data**. They're two completely separate concerns.

| Concern                       | What solves it               |
| ----------------------------- | ---------------------------- |
| Too many DOM nodes            | Virtualization               |
| Too much data fetched upfront | Infinite scroll (pagination) |

Infinite scroll watches how far the user has scrolled through the _currently loaded_ data. When they're close to the bottom, it fires an API request for the next page.

```
API has 600 items total

вЖТ Fetch page 1 (items 1вАУ20)
   Virtualization renders ~10 DOM nodes from those 20

вЖТ User scrolls, approaching item 17
   Trigger: fetch page 2 (items 21вАУ40)
   Now 40 items in memory, still ~10 DOM nodes in the DOM

вЖТ User scrolls, approaching item 37
   Trigger: fetch page 3 (items 41вАУ60)
   60 items in memory. DOM still lean.

вЖТ User scrolls back up to item 5
   Zero API calls. Data is in memory.
```

The JS array **only ever grows**. The DOM stays small throughout.

Combine both and you get a list that feels instant to scroll, never fetches more than it needs to, and handles thousands of items without breaking a sweat.

### Understanding `useVirtualizer`

The main hook is `useVirtualizer`. Let's go through its options and map each one back to what we discussed:

```ts
const virtualizer = useVirtualizer({
  // Total item count вАФ this is what determines the phantom container height
  count: items.length,

  // The DOM element that actually scrolls
  // Must have overflow-y: auto or scroll
  getScrollElement: () => scrollRef.current,

  // Height estimate per item вАФ used for phantom height and initial offset math
  // For uniform card heights, this can be exact
  estimateSize: () => 72,

  // Buffer вАФ extra items to render above and below the visible area
  // Higher = smoother scroll, more DOM nodes
  // 5 is a good default for most lists
  overscan: 5,
});
```

And what comes back:

```ts
// Items that should be in the DOM right now
const items = virtualizer.getVirtualItems();
// e.g. [{ index: 40, start: 2880, size: 72, key: 40 }, ...]
//   index вЖТ position in your data array
//   start вЖТ translateY value (where to place it)

// The phantom container height
const totalHeight = virtualizer.getTotalSize();
// e.g. 36000px
```

> **вЪ†пЄП `getScrollElement` is the most important option to get right.** It must point to the element with `overflow-y: scroll` or `auto` вАФ not a parent, not `document.body`, but the actual scrolling container. If this is wrong, the virtualizer's scroll calculations will be off.

### A Plain Virtualized List

```tsx
import { useRef } from 'react';
import { useVirtualizer } from '@tanstack/react-virtual';

type Item = {
  id: string;
  title: string;
  description: string;
};

type VirtualListProps = {
  items: Item[];
};

export const VirtualList = ({ items }: VirtualListProps) => {
  const scrollRef = useRef<HTMLDivElement>(null);

  const virtualizer = useVirtualizer({
    count: items.length,
    getScrollElement: () => scrollRef.current,
    estimateSize: () => 72,
    overscan: 5,
  });

  return (
    // Fixed height scroll container
    <div ref={scrollRef} style={{ height: '600px', overflowY: 'auto' }}>
      {/* Phantom container вАФ full list height */}
      <div style={{ height: virtualizer.getTotalSize(), position: 'relative' }}>
        {virtualizer.getVirtualItems().map((virtualItem) => (
          // Absolutely positioned at its exact offset
          <div
            key={virtualItem.key}
            style={{
              position: 'absolute',
              top: 0,
              left: 0,
              width: '100%',
              // transform instead of top for GPU compositing
              transform: `translateY(${virtualItem.start}px)`,
            }}
          >
            <ResultCard item={items[virtualItem.index]} />
          </div>
        ))}
      </div>
    </div>
  );
};
```

> **вЪ†пЄП `transform` vs `top`:** We use `transform: translateY(${virtualItem.start}px)` rather than `top: ${virtualItem.start}px`. Transforms are composited on the GPU and don't trigger layout recalculations. Slightly but meaningfully better for scroll performance вАФ which is exactly the thing we're trying to improve here.

### Adding Infinite Scroll

The trigger logic is straightforward вАФ watch the last visible item index and compare it against the total fetched count:

```ts
const THRESHOLD = 5; // fetch when within 5 items of the bottom

const lastVisibleItem = virtualItems[virtualItems.length - 1];

if (
  lastVisibleItem &&
  lastVisibleItem.index >= items.length - THRESHOLD &&
  hasNextPage &&
  !isFetchingNextPage
) {
  fetchNextPage();
}
```

Here's the full component with both concerns wired together:

```tsx
import { useEffect, useRef } from 'react';
import { useVirtualizer } from '@tanstack/react-virtual';

type PaginationConfig = {
  hasNextPage: boolean;
  isFetchingNextPage: boolean;
  onFetchNextPage: () => void;
};

type InfiniteVirtualListProps = {
  items: Item[];
  paginationConfig?: PaginationConfig; // optional вАФ plain list if not passed
};

export const InfiniteVirtualList = ({
  items,
  paginationConfig,
}: InfiniteVirtualListProps) => {
  const scrollRef = useRef<HTMLDivElement>(null);

  const virtualizer = useVirtualizer({
    count: items.length,
    getScrollElement: () => scrollRef.current,
    estimateSize: () => 72,
    overscan: 5,
  });

  const virtualItems = virtualizer.getVirtualItems();

  // Infinite scroll trigger
  useEffect(() => {
    if (!paginationConfig) return;

    const { hasNextPage, isFetchingNextPage, onFetchNextPage } =
      paginationConfig;
    const lastItem = virtualItems[virtualItems.length - 1];

    if (!lastItem) return;

    if (
      lastItem.index >= items.length - 5 &&
      hasNextPage &&
      !isFetchingNextPage
    ) {
      onFetchNextPage();
    }
  }, [virtualItems, items.length, paginationConfig]);

  return (
    <div ref={scrollRef} style={{ height: '600px', overflowY: 'auto' }}>
      <div style={{ height: virtualizer.getTotalSize(), position: 'relative' }}>
        {virtualItems.map((virtualItem) => (
          <div
            key={virtualItem.key}
            style={{
              position: 'absolute',
              top: 0,
              left: 0,
              width: '100%',
              transform: `translateY(${virtualItem.start}px)`,
            }}
          >
            <ResultCard item={items[virtualItem.index]} />
          </div>
        ))}
      </div>

      {/* Loading indicator below the list */}
      {paginationConfig?.isFetchingNextPage && (
        <div style={{ padding: '16px', textAlign: 'center' }}>
          Loading more...
        </div>
      )}
    </div>
  );
};
```

> **вЪ†пЄП `paginationConfig` is optional by design.** If you don't pass it, you get a plain virtualized list вАФ no infinite scroll, no API triggers. If you pass it, infinite scroll activates. This keeps the component clean for consumers who don't need pagination, without any breaking changes.

## What About Variable Height Items?

Everything above assumes each item is the same height. If your cards vary вАФ say some have an extra line of text or an optional image вАФ use `measureElement` to let TanStack Virtual measure each item after it renders:

```tsx
const virtualizer = useVirtualizer({
  count: items.length,
  getScrollElement: () => scrollRef.current,
  estimateSize: () => 80, // rough estimate, doesn't need to be exact
  overscan: 5,
});

// In your item render:
{
  virtualItems.map((virtualItem) => (
    <div
      key={virtualItem.key}
      // Pass ref + data-index so the virtualizer can measure this element
      ref={virtualizer.measureElement}
      data-index={virtualItem.index}
      style={{
        position: 'absolute',
        top: 0,
        left: 0,
        width: '100%',
        transform: `translateY(${virtualItem.start}px)`,
      }}
    >
      <ResultCard item={items[virtualItem.index]} />
    </div>
  ));
}
```

The virtualizer measures the rendered height of each item and builds an offset map. The phantom container height adjusts dynamically as items are measured. For most card-based UIs with consistent structure, you won't need this вАФ but it's worth knowing it exists.

## TanStack Virtual вАФ Quick Reference

**`useVirtualizer` options:**

| Option             | Type                        | What it does                                  |
| ------------------ | --------------------------- | --------------------------------------------- |
| `count`            | `number`                    | Total items вАФ drives phantom container height |
| `getScrollElement` | `() => HTMLElement \| null` | The scrolling container                       |
| `estimateSize`     | `() => number`              | Item height estimate in px                    |
| `overscan`         | `number`                    | Buffer items outside the viewport             |
| `measureElement`   | ref callback                | For variable/dynamic item heights             |

**What comes back:**

| Property            | What it gives you          |
| ------------------- | -------------------------- |
| `getVirtualItems()` | Items to render right now  |
| `getTotalSize()`    | Phantom container height   |
| `VirtualItem.index` | Index into your data array |
| `VirtualItem.start` | `translateY` value         |
| `VirtualItem.key`   | Stable React key           |

## Putting It All Together

Here's the mental model that makes everything click:

```
Your API вЖТ JS array (grows with each page)
               вЖУ
    Virtualization reads the array
               вЖУ
    Keeps ~10 DOM nodes alive at any time
               вЖУ
    User scrolls near bottom of loaded data
               вЖУ
    Infinite scroll вЖТ API call вЖТ array grows
               вЖУ
    Back to top. DOM stays lean throughout.
```

DOM nodes are cheap when there are 10 of them. They're expensive when there are 500. Virtualization is the difference between the two вАФ and infinite scroll is what makes sure the data side never gets ahead of what the user actually needs.

That flame graph I was staring at before? Completely different story once both were in place. The scroll was smooth, the memory stayed flat, and the browser stopped complaining.

If you're building any kind of long list with paginated data, you almost certainly need both of these working together. The good news is they're simpler to implement than they look вАФ once you understand what each one is actually doing.
