## Knowing vs. Shipping

There's a version of this post where I pretend I'd never heard of virtualization before some senior engineer casually dropped it in sprint planning and I nodded confidently while internally panicking. That version is more dramatic. It's also not true.

I'd implemented virtualization before. Understood the phantom container trick, had shipped infinite scroll with Apollo, knew what `useVirtualizer` did. Confident enough that when a similar requirement landed recently, I figured it would be a clean, familiar implementation.

Production had a different opinion. 😅

What I didn't account for were the edge cases that only surface when real data meets real layout complexity — cards with inconsistent heights, panels opening mid-scroll, cache policies that silently swallow your pagination without a single error in the console. None of that lives in any tutorial I've read.

So this post has two parts. First is the concept — clean, quick, linking out where I've already written the deep dive. Second is what actually happened when we shipped it. The stuff that only production teaches you.

---

## First — A Quick Name Thing

Virtualization and windowing are the same thing. You will see both terms scattered across docs, blog posts, and Stack Overflow answers depending on who wrote them. TanStack calls their library `@tanstack/react-virtual`. React's own documentation uses the word "windowing." Same technique, different vocabulary. Now we're aligned and you won't get confused mid-googling.

---

## The Two Concerns You Need To Separate

Before anything else, this distinction is the most important thing in this post:

**Virtualization manages the DOM. Infinite scroll manages the data.**

They are completely separate concerns that happen to work beautifully together.

| Concern                       | What solves it               |
| ----------------------------- | ---------------------------- |
| Too many DOM nodes            | Virtualization               |
| Too much data fetched upfront | Infinite scroll (pagination) |

If you want to understand _why_ pagination matters — offset vs cursor, when each breaks, how Apollo caches pages — I wrote that in detail [here](https://manuj-sankrit-8055.vercel.app/blog/offset-vs-cursor-pagination). This post assumes you've got that context and focuses entirely on the rendering side.

## The Lie The Scroll Tells

When you render 500 items in React, you create 500 real DOM nodes. Each one has memory, layout cost, paint cost, scroll overhead. The browser faithfully tracks all of them on every frame — even the 480 that are nowhere near the viewport.

Virtualization fixes this with an elegant deception.

**Only create DOM nodes for items inside the visible viewport.** Everything else simply doesn't exist in the DOM.

But if you only render 12 items, the scrollbar would behave like there are only 12 items. The whole illusion falls apart. So virtualization maintains the lie with two tricks:

**Trick 1 — The Phantom Container**

The outer wrapper gets an explicit height equal to what the full list _would_ be if every item was rendered:

```
Total height = itemCount × itemHeight = 500 × 72px = 36,000px
```

This invisible box makes the scrollbar behave correctly. The user's scroll muscle memory stays intact. They have no idea only 12 DOM nodes are alive.

**Trick 2 — Absolute Positioning**

Rendered items sit absolutely inside that phantom container, placed at the exact pixel offset where they should visually appear. As the user scrolls, the library runs a tight loop — read `scrollTop`, calculate which items should be visible, mount those DOM nodes, position them, unmount the rest.

```
User scrolls → scrollTop = 2900px
startIndex = Math.floor(2900 / 72) = 40
→ Render items 37–53
→ ~15 DOM nodes instead of 500
```

This runs on every scroll event, which is why it has to be fast. And why the virtualizer's internals are carefully optimised — you don't need to worry about that part, but you do need to feed it accurate information. More on that shortly.

## The Implementation

```bash
npm install @tanstack/react-virtual
```

The core hook:

```ts
const virtualizer = useVirtualizer({
  count: items.length, // drives phantom container height
  getScrollElement: () => scrollRef.current, // must point to the element with overflow-y scroll
  estimateSize: () => 72, // item height estimate in px
  overscan: 5, // buffer items above and below viewport
});
```

What comes back:

```ts
virtualizer.getVirtualItems(); // which items to render right now
virtualizer.getTotalSize(); // phantom container height
// Each virtual item: { index, start, size, key }
```

The render structure:

```tsx
<div ref={scrollRef} style={{ height: '600px', overflowY: 'auto' }}>
  <div style={{ height: virtualizer.getTotalSize(), position: 'relative' }}>
    {virtualizer.getVirtualItems().map((virtualItem) => (
      <div
        key={virtualItem.key}
        style={{
          position: 'absolute',
          top: 0,
          left: 0,
          width: '100%',
          transform: `translateY(${virtualItem.start}px)`, // GPU compositing > top offset
        }}
      >
        <ResultCard item={items[virtualItem.index]} />
      </div>
    ))}
  </div>
</div>
```

> **⚠️ `getScrollElement` is the most important option to get right.** It must point to the element with `overflow-y: scroll` or `auto` — not a parent wrapper, not `document.body`, the actual scrolling container. Wrong element means broken scroll calculations, and the virtualizer gives you no helpful error about it.

Wiring in infinite scroll is a `useEffect` watching the last visible item:

```ts
useEffect(() => {
  const lastItem = virtualItems[virtualItems.length - 1];
  if (!lastItem) return;

  if (
    lastItem.index >= items.length - 5 &&
    hasNextPage &&
    !isFetchingNextPage
  ) {
    fetchNextPage();
  }
}, [virtualItems, items.length]);
```

That's the concept. Clean, composable, works great in a sandbox.

## What Production Actually Taught Me

We shipped this in a real app — a shared component library consumed by multiple teams, with Apollo handling server-side offset pagination. Here is what we did not see coming.

### 1. Your `estimateSize` Is Probably Wrong. By a Lot.

Every example online passes something neat like `72` or `80` to `estimateSize`. Real cards are not neat.

Our initial estimate was `98px`. Real rendered height was `~162px` — because of badges, variable address lengths, and padding that looks small on a design file and isn't. The gap between estimate and reality causes jitter on initial render as the virtualizer corrects its position table item by item. Not a crash, not a console error — just a slightly janky first impression.

The fix: measure a few real rendered cards, bump your estimate _up_ rather than down when guessing, and use `measureElement` for genuinely dynamic heights:

```tsx
<div
  ref={virtualizer.measureElement}
  data-index={virtualItem.index}
  style={{
    position: 'absolute',
    top: 0,
    left: 0,
    width: '100%',
    transform: `translateY(${virtualItem.start}px)`,
  }}
>
  <ResultCard item={items[virtualItem.index]} />
</div>
```

The virtualizer measures each card's real height after it renders and builds an accurate offset table. Jitter disappears.

### 2. `measureElement` Can Return 0. And It Will Silently Corrupt Everything.

This one cost us real debugging time.

When a details panel opened alongside the list, it triggered a layout recalculation. During that recalculation, `ResizeObserver` callbacks fired and `getBoundingClientRect()` returned `0` for some items — not an error, just `0`. The virtualizer dutifully wrote those zeroes into its position cache. Subsequent items stacked on top of each other at position `0`.

The list looked like a car crash.

The fix is a single guard in your `measureElement` callback:

```ts
// If the measured height comes back as 0, fall back to estimated height
if (measuredHeight === 0) return estimatedItemHeight;
```

One line. Not something you'd think to add until production bites you.

### 3. Cards Need Uniform Height. More Than You Think.

Our result cards had an optional badges row. Cards with badges rendered taller than cards without. Inconsistent heights meant the virtualizer's measurements were slightly different for every card, which accumulated into misalignment errors mid-scroll — items drifting out of place the further down the list you went.

The fix that actually worked: an invisible placeholder `div` matching the badge row height on cards that have no badges.

```tsx
{
  badges.length > 0 ? (
    <BadgeRow badges={badges} />
  ) : (
    <div aria-hidden='true' style={{ height: '24px' }} /> // invisible spacer
  );
}
```

All cards are now uniformly tall regardless of content. Virtualizer measurements stabilise. `aria-hidden="true"` keeps screen readers from stumbling over an empty div. It's the kind of solution that looks bizarre in a PR review and makes complete sense the moment you know the why.

### 4. The Scroll Container Needs `flex: 1, minHeight: 0`. Trust Me.

The virtualizer needs a bounded height to scroll inside. If its parent container can grow freely, the scroll container expands to fit all content — and you effectively have no virtualization at all, just a very elaborate static list.

```css
.list-container {
  flex: 1;
  min-height: 0; /* 👈 this is the one */
  overflow: hidden;
}
```

`min-height: 0` overrides the default `min-height: auto` on flex children that prevents them from shrinking below their content height. Without it, `overflow: auto` on the scroll container never triggers. This will take you an hour to find and one line to fix.

### 5. Apollo's Cache Will Silently Eat Your Pages.

By default, Apollo replaces the cached result entirely on every `fetchMore` call. Page 1 loads — great. Page 2 fetches — page 1 disappears. The list silently resets to 20 items. No error, no warning, no indication anything is wrong.

The fix is a custom type policy that tells Apollo to accumulate pages rather than replace them:

```ts
results: {
  keyArgs: ['location', 'filters'],   // what makes this query unique in the cache
  merge(existing, incoming, { args }) {
    if (!existing || args?.offset === 0) {
      return incoming;  // fresh search: replace entirely
    }
    return {
      ...incoming,
      items: [...existing.items, ...incoming.items],  // paginating: append
    };
  },
}
```

The `offset === 0` guard is important. When the user changes their search, you want a complete cache reset — not the new results appended to the previous ones. Without it, dealers from Mumbai silently stack on top of dealers from Delhi and everyone is confused.

For a deeper understanding of why offset pagination works this way (and when cursor-based pagination is the better call), [that post covers it in full](https://manuj-sankrit-8055.vercel.app/blog/offset-vs-cursor-pagination).

### 6. Stale Closures Will Find You.

Inside a `scrollToIndex` effect, we referenced the items array directly from the closure. When a new page loaded, the effect still held a reference to the old array. Scroll positions were calculated against stale data, and the virtualizer jumped to the wrong item.

```ts
// Problem — locations is stale by the time the effect runs
useEffect(() => {
  virtualizer.scrollToIndex(locations.findIndex((l) => l.id === selectedId));
}, [selectedId]);

// Fix — ref stays current
const locationsRef = useRef(locations);
useEffect(() => {
  locationsRef.current = locations;
}, [locations]);

useEffect(() => {
  virtualizer.scrollToIndex(
    locationsRef.current.findIndex((l) => l.id === selectedId),
  );
}, [selectedId]);
```

Stale closures in effects are not new territory. But they are particularly sneaky inside virtualizer logic because the symptom — wrong scroll position — looks like a virtualizer bug rather than a React hooks bug. You spend time reading TanStack docs when the problem is in your own effect.

## The Mental Model That Makes It Click

```
API → JS array (grows with each page, never shrinks)
           ↓
  Virtualization reads the array
           ↓
  Keeps ~15 DOM nodes alive at any time
           ↓
  User scrolls near bottom of loaded data
           ↓
  Infinite scroll → API call → array grows
           ↓
  Back to top. DOM stays lean throughout.
```

JS array grows. DOM stays flat. That is the whole game.

The concept takes an afternoon to understand. The sandbox takes a day to build. Production takes longer still — not because the implementation is hard, but because real data surfaces edge cases that a clean demo never replicates. The card height inconsistencies, the panel interactions, the cache behavior, the CSS gotcha that silently disables your scroll container.

You now know what to look for before you go looking.
